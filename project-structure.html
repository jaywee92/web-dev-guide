🏭 App Factory Pattern

Erstellt die Flask-App dynamisch mit verschiedenen Konfigurationen. Ermöglicht Testing und Multi-Instanz-Deployment.

app/__init__.py

Python 
from flask import Flask
from flask_sqlalchemy import SQLAlchemy

db = SQLAlchemy()

def create_app(config_name='default'):
    app = Flask(__name__)
    app.config.from_object(
        config[config_name]
    )

    # Init Extensions
    db.init_app(app)

    # Register Blueprints
    from .routes.auth import auth_bp
    from .routes.api import api_bp

    app.register_blueprint(auth_bp)
    app.register_blueprint(api_bp)

    return app

run.py

Python 
import os
from app import create_app, db

config_name = os.getenv(
    'FLASK_ENV',
    'default'
)
app = create_app(config_name)

with app.app_context():
    db.create_all()

if __name__ == '__main__':
    app.run(
        debug=True,
        host='0.0.0.0'
    )
✨ Vorteile des App Factory Pattern
  • Testbar – Mehrere App-Instanzen für Tests
  • Konfigurierbar – Unterschiedliche Settings pro Umgebung
  • Erweiterbar – Blueprints einfach hinzufügbar
  • Wartbar – Klare Trennung von Concerns

📋 Blueprints – Modulare Routes

Blueprints ermöglichen es, Routes in separate Module zu organisieren – ideal für große Anwendungen.

app/routes/auth.py

Python 
from flask import Blueprint

auth_bp = Blueprint(
    'auth',
    __name__,
    url_prefix='/auth'
)

@auth_bp.route('/login', methods=['POST'])
def login():
    return {'msg': 'Login erfolgreich'}

@auth_bp.route('/logout')
def logout():
    return {'msg': 'Logout erfolgreich'}

app/routes/api.py

Python 
from flask import Blueprint, jsonify
from app.models import User

api_bp = Blueprint(
    'api',
    __name__,
    url_prefix='/api'
)

@api_bp.route('/users')
def get_users():
    users = User.query.all()
    return jsonify([
        u.to_dict() for u in users
    ])
🎯 Blueprint-Routing
  • auth_bp → /auth/login und /auth/logout
  • api_bp → /api/users und weitere API-Routes
  • Blueprints werden in create_app() registriert

⚙️ Konfiguration

Unterschiedliche Settings für Development, Production und Testing – ohne Code-Änderungen.

config.py – Konfigurationsklassen

Python 
import os

class Config:
    """Basis-Konfiguration"""
    SECRET_KEY = os.getenv('SECRET_KEY', 'dev-key')
    SQLALCHEMY_TRACK_MODIFICATIONS = False

class DevelopmentConfig(Config):
    """Development-Einstellungen"""
    DEBUG = True
    TESTING = False
    SQLALCHEMY_DATABASE_URI = 'sqlite:///dev.db'

class ProductionConfig(Config):
    """Production-Einstellungen"""
    DEBUG = False
    SQLALCHEMY_DATABASE_URI = os.getenv('DATABASE_URL')

class TestingConfig(Config):
    """Testing-Einstellungen"""
    TESTING = True
    SQLALCHEMY_DATABASE_URI = 'sqlite:///:memory:'

config = {
    'development': DevelopmentConfig,
    'production': ProductionConfig,
    'testing': TestingConfig,
    'default': DevelopmentConfig
}

🔒 Virtual Environment (venv)

Isoliert Python-Pakete pro Projekt – verhindert Konflikte zwischen Projekten.

❌ Ohne venv (Gefährlich)
System Python
site-packages/
flask==2.0
requests==2.25
⚠️ Alle Projekte teilen sich Pakete – Versions-Konflikte!
✅ Mit venv (Sicher)
venv/lib/python3.9/
flask==2.3
requests==2.28
✓ Isolierte, projektspezifische Pakete

venv Setup & Workflow

Bash – Terminal Commands 
# 1. Virtuelle Umgebung erstellen
python -m venv venv

# 2. venv aktivieren
source venv/bin/activate  # Linux/Mac
venv\Scripts\activate        # Windows

# 3. Abhängigkeiten installieren
pip install flask flask-sqlalchemy python-dotenv

# 4. Abhängigkeiten frieren (speichern)
pip freeze > requirements.txt

# 5. Später: Abhängigkeiten wiederherstellen
pip install -r requirements.txt

# 6. venv deaktivieren
deactivate
📦 requirements.txt Beispiel 
Flask==2.3.2
Flask-SQLAlchemy==3.0.3
python-dotenv==1.0.0
gunicorn==20.1.0

✅ Zusammenfassung – Best Practices

🏗️ Struktur
  • Nutze App Factory für Flexibilität
  • Blueprints für modulare Routes
  • Trennung: app/, config/, templates/, static/
🔒 Sicherheit
  • .env für Geheimnisse (nicht ins Git!)
  • venv für Abhängigkeiten
  • .gitignore für sensitive Dateien
⚙️ Konfiguration
  • config.py mit Development/Production/Testing
  • Umgebungsvariablen via .env
  • app.config.from_object() zum Laden
📦 Abhängigkeiten
  • requirements.txt mit pip freeze
  • Versionsnummern festlegen
  • pip install -r requirements.txt zum Restore
🧪 Testing
  • create_app() mit test config
  • In-memory SQLite für Tests
  • Isolierte Test-Umgebung
🚀 Deployment
  • ProductionConfig mit echter DB
  • DEBUG=False in Production
  • gunicorn als WSGI Server
