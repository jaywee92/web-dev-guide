Flask Architektur

Verstehen Sie, wie Flask intern funktioniert

🔗 WSGI Layer

Flask sitzt auf der Werkzeug WSGI-Implementierung. Dies ist die Schicht zwischen dem Webserver und Ihrer Anwendung.

🌐 Browser
↓
⚙️ Webserver (Gunicorn / nginx)
↓
🔄 WSGI Interface (Werkzeug)
↓
🐍 Flask Application
Was ist WSGI? Web Server Gateway Interface (WSGI) ist ein Standard-Interface zwischen Webservern und Python-Webanwendungen. Werkzeug, die Bibliothek hinter Flask, implementiert dieses Interface.
# Flask nutzt Werkzeug's WSGI-Implementierung from werkzeug.serving import run_simple app = Flask(__name__) # Unter der Haube: app ist ein WSGI callable run_simple('127.0.0.1', 5000, app)

🔀 Request / Response Zyklus

So verarbeitet Flask eine HTTP-Anfrage von Anfang bis Ende:

🌐
1. HTTP Request
Client sendet eine HTTP-Anfrage an den Server
→
📦
2. Request-Objekt
Flask erstellt ein Request-Objekt mit allen Daten
🗺️
3. URL Routing
Werkzeug Router matcht die URL mit einer Route
→
⚡
4. View Function
Die entsprechende View-Funktion wird aufgerufen
📝
5. Response erstellen
Die View gibt Daten zurück, Flask erstellt Response
→
✉️
6. HTTP Response
Response wird zum Browser des Clients zurückgesendet
@app.route('/user/<int:user_id>') def get_user(user_id): # Flask ruft diese Funktion auf wenn die Route matcht return jsonify({'id': user_id}) # Wenn Browser /user/123 anfagt: # 1. WSGI erhält Request # 2. Flask Route '/user/<int:user_id>' matcht # 3. get_user(123) wird aufgerufen # 4. Response wird erstellt und zurückgesendet

⚙️ Flask App Internals

Diese Komponenten machen eine Flask-Anwendung aus:

🐍
app = Flask(__name__)
Das zentrale Flask-Anwendungsobjekt. Speichert alle Konfigurationen und Routinen.
⚙️
Config
Konfigurationseinstellungen wie DEBUG, SECRET_KEY, Datenbankverbindungen.
🗺️
URL Map
Werkzeug Map mit allen registrierten Routen und ihren Regeln.
⚡
View Functions
Die Funktionen, die bei einer bestimmten Route aufgerufen werden.
🚨
Error Handlers
Spezielle Funktionen für 404, 500 und andere HTTP-Fehler.
📄
Template Engine
Jinja2 Engine für das Rendering von HTML-Templates.
📁
Static Files
CSS, JavaScript, Bilder und andere statische Dateien.
🧩
Blueprints
Modularisierung der Anwendung in wiederverwendbare Komponenten.
from flask import Flask app = Flask(__name__) # Config app.config['DEBUG'] = True app.config['SECRET_KEY'] = 'secret' # URL Map (automatisch mit @app.route erstellt) @app.route('/') def home(): return 'Hello World' # Error Handler @app.errorhandler(404) def not_found(error): return 'Page not found', 404

🧩 Blueprints & Modularisierung

Blueprints ermöglichen es, eine Flask-App in wiederverwendbare Module zu unterteilen:

🐍 Main Flask App
🔐
auth
Login, Registrierung, Sessions
📡
api
REST API Endpoints
👑
admin
Verwaltungspanel
# auth/blueprint.py from flask import Blueprint auth_bp = Blueprint('auth', __name__) @auth_bp.route('/login', methods=['POST']) def login(): return jsonify({'status': 'logged in'}) # app.py app = Flask(__name__) app.register_blueprint(auth_bp, url_prefix='/auth')
Vorteil: Blueprints ermöglichen saubere Code-Organisation, Wiederverwendbarkeit und bessere Testbarkeit. Sie sind ideal für größere Projekte mit mehreren Modulen.

🔄 Middleware & Request Lifecycle

Flask führt Code an spezifischen Punkten im Request-Lifecycle aus:

before_request
→
View Function
→
after_request

Alle Requests gehen durch:

1️⃣ before_request()
→
2️⃣ request()
→
3️⃣ teardown_request()
before_request
Wird vor jeder View-Funktion ausgeführt. Ideal für Authentifizierung, Rate Limiting, Logging.
after_request
Wird nach der View-Funktion aber vor der Response ausgeführt. Ideal für Response-Modifikation, Header setzen.
teardown_request
Wird am Ende des Request aufgerufen, egal ob Error oder nicht. Ideal für Cleanup.
errorhandler
Behandelt Exceptions. Wird aufgerufen wenn ein Fehler im Request auftritt.
@app.before_request def before_request(): # Authentifizierung prüfen if current_user is None: return redirect('/login') @app.after_request def after_request(response): # CORS Headers hinzufügen response.headers['Access-Control-Allow-Origin'] = '*' return response @app.teardown_request def teardown(exception=None): # Datenbankverbindung schließen if db is not None: db.close()

🚀 Weitere wichtige Konzepte

📦
Request Context
Flask erstellt einen Request Context für jeden Request. Dies ermöglicht globalen Zugriff auf Request-Daten wie request.args, request.form.
🔐
Application Context
Der Application Context speichert App-Level Daten. Wird beim App Start erstellt und beim Shutdown zerstört.
🎣
Hooks & Signals
before_first_request, before_request, after_request, teardown_request - definieren den Request-Lifecycle.
⚡
Extensions
Flask-SQLAlchemy, Flask-Login, Flask-WTF - erweitern Flask mit zusätzlicher Funktionalität.
🔗
URL Building
url_for() generiert URLs basierend auf View-Funktionsnamen. Ermöglicht sichere URL-Erzeugung.
📝
Werkzeug Utilities
Flask nutzt Werkzeug für Request Handling, Routing, Sessions, Cookies und HTTP Utilities.
Context Isolation: Flask isoliert jeden Request in seinen eigenen Context. Das bedeutet, dass Variablen wie request, g und session immer auf den aktuellen Request bezogen sind, ohne dass Sie diese als Parameter übergeben müssen. Dies ist eines der mächtigsten Features von Flask!
# URL Building @app.route('/user/<int:user_id>') def user_profile(user_id): return render_template('profile.html') # In Template oder Python Code: url_for('user_profile', user_id=42) # → /user/42 # Context Usage from flask import request, g @app.route('/api/data') def api_data(): user_id = request.args.get('user_id') # Request Context g.user_data = fetch_user(user_id) # Request-scoped storage return jsonify(g.user_data)
