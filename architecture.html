Flask Architecture

Understand how Flask works internally

🔗 WSGI Layer

Flask sits on top of Werkzeug's WSGI implementation. This is the layer between the web server and your application.

🌐 Browser
↓
⚙️ Webserver (Gunicorn / nginx)
↓
🔄 WSGI Interface (Werkzeug)
↓
🐍 Flask Application
What is WSGI? Web Server Gateway Interface (WSGI) is a standard interface between web servers and Python web applications. Werkzeug, the library behind Flask, implements this interface.

🔀 Request / Response Cycle

Here's how Flask processes an HTTP request from start to finish:

🌐
1. HTTP Request
Client sends an HTTP request to the server
→
📦
2. Request Object
Flask creates a Request object with all data
🗺️
3. URL Routing
Werkzeug router matches the URL with a route
→
⚡
4. View Function
The corresponding view function is called
📝
5. Response Created
The view returns data, Flask creates the Response
→
✉️
6. HTTP Response
Response is sent back to the client's browser

# auth/blueprint.py
from flask import Blueprint
auth_bp = Blueprint('auth', '__name__')
@auth_bp.route('/login', methods=['POST'])
def login():
return jsonify({'status': 'logged in'})
# app.py
app = Flask('__main__')
app.register_blueprint(auth_bp, url_prefix='/auth')
Advantage: Blueprints enable clean code organization, reusability, and better testability. They are ideal for larger projects with multiple modules.

🔄 Middleware & Request Lifecycle

Flask runs code at specific points in the request lifecycle:

before_request
→
View Function
→
after_request

All requests go through:

1️⃣ before_request()
→
2️⃣ request()
→
3️⃣ teardown_request()
before_request
Runs before every view function. Ideal for authentication, rate limiting, logging.
after_request
Runs after the view function but before the response. Ideal for response modification, setting headers.
teardown_request
Called at the end of the request, regardless of errors. Ideal for cleanup.
errorhandler
Handles exceptions. Called when an error occurs in the request.

def before_request():
# Check authentication

@app.after_request
def after_request(response):
# Add CORS headers
response.headers['Access-Control-Allow-Origin'] = '*'
return response
@app.teardown_request
def teardown(exception=None):
# Close database connection
if db is not None:
db.close()

🚀 Advanced Concepts

📦
Request Context
Flask creates a Request Context for each request, enabling global access to request data like request.args, request.form.
🔐
Application Context
The Application Context stores app-level data. Created on app start, destroyed on shutdown.
🎣
Hooks & Signals
before_first_request, before_request, after_request, teardown_request — define the request lifecycle.
⚡
Extensions
Flask-SQLAlchemy, Flask-Login, Flask-WTF — extend Flask with additional functionality.
🔗
URL Building
url_for() generates URLs based on view function names. Enables safe URL generation.
📝
Werkzeug Utilities
Flask uses Werkzeug for request handling, routing, sessions, cookies, and HTTP utilities.
Context Isolation: Flask isolates each request in its own context. This means variables like request, g, and session always refer to the current request without passing them as parameters. This is one of Flask's most powerful features!

# Context usage
from flask import request, g
@app.route('/api/data')
def api_data():
user_id = request.args.get('user_id') # Request context
g.user_data = fetch_user(user_id) # Request-scoped storage
return jsonify(g.user_data)
